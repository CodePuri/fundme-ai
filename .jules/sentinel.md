## 2026-04-20 - [CRITICAL] Prevent Supabase Error Message Leakage
**Vulnerability:** The `app/api/onboarding/route.ts` API endpoints were directly returning `error.message` from failed Supabase database operations to the client in HTTP 500 responses. This constitutes information leakage, potentially exposing internal database structure, schema details, or other sensitive backend information to an attacker.
**Learning:** Returning detailed error messages directly to the client without sanitization is a common anti-pattern that violates the "Fail securely" principle. While useful for debugging, it provides reconnaissance information to adversaries. The application lacks a centralized error handler for API routes, increasing the risk of this pattern appearing in other endpoints.
**Prevention:** Always log detailed error objects on the server-side (e.g., using `console.error` or a logging library) for operational monitoring. Return generic, non-descriptive error messages (e.g., "An internal server error occurred") to the client. This ensures the client knows a failure occurred without knowing the sensitive specifics of *why* or *how* it failed.
