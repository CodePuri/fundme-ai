## 2026-04-21 - Fix Information Leakage and Add Input Validation
**Vulnerability:** The `/api/onboarding` API returned direct Supabase error objects in error responses, risking information leakage about the database schema or internals. The API also completely lacked input validation, which could allow users to insert arbitrarily large payloads causing DoS or insert unsafe URLs.
**Learning:** Due to the absence of standard validation libraries (e.g. `zod`), all input constraint checks and validations (such as type, length, and URL protocols) must be manually written. Further, since exceptions and database errors can be verbose, you must explicitly intercept errors, log them on the server securely, and mask them behind a generic error message for the client.
**Prevention:** Always log detailed errors to server logs (e.g., `console.error`) and return a generic 500 error (`"An internal server error occurred"`) to the client. Validate the length of string fields (e.g., maximum limits), and enforce strict URL protocol rules (`http:`, `https:`) using manual conditionals when libraries are unavailable.
